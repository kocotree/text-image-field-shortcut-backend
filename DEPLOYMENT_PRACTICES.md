# Deployment Practices

这份文档是给当前项目预留的上线实践清单，针对的场景是：

- Python 后端
- Flask 起步
- 链路包含模型调用、文件下载、OSS 上传
- 目标吞吐至少每分钟 100 次请求

当前先作为后续实施参考，不要求一次性全部落地。

## 结论

每分钟 100 次请求本身不高，平均约为 `1.67 RPS`。  
真正要关注的是这些问题：

- 突发并发是否会集中出现
- 单个请求是否包含长时间外部 IO
- 模型调用和 OSS 上传是否会拖住 worker
- 文件是否会占用过多内存

所以当前项目的重点不是“Python 扛不扛得住”，而是把外部调用、文件处理、超时和隔离做好。

## 当前项目的优先级建议

### 1. 不要使用 Flask 开发服务器直接上线

当前本地用：

```powershell
uv run python main.py
```

这只适合开发调试。

后续上线建议改为生产级 WSGI/ASGI 启动方式，例如：

- `gunicorn`
- 或后续迁移到 `uvicorn` / `FastAPI`

如果短期继续保持 Flask，建议优先考虑：

- `gunicorn + gevent`
- 或 `gunicorn + 多 worker`

## 2. 统一管理外部请求超时

当前项目后面会依赖这些外部 IO：

- 调 Maibao
- 下载输入文件 URL
- 上传 OSS

这些操作都必须设置超时，否则少量卡死请求就可能拖垮整个实例。

建议：

- 连接超时和读超时分开配置
- 不允许无限等待
- 所有外部调用统一封装

## 3. 复用 HTTP 连接，不要每次新建

如果后续改成通过 HTTP 调 Maibao、下载文件、调用内部服务：

- 使用长期复用的 HTTP client / session
- 避免每个请求都新建连接池

这样能减少握手开销，也能提升并发稳定性。

## 4. 文件处理尽量流式，不要默认整文件入内存

当前项目后续会支持两类输入：

- 文件 URL
- 文件流

后续实现时建议：

- 下载时尽量流式读取
- 上传 OSS 时尽量流式传输
- 对大文件不要默认整块读入内存

如果短期为了开发速度先用内存方式，也建议后续尽快替换。

## 5. 服务层要继续拆清楚

当前 backend 已经开始按服务层拆分。后续建议保持下面的职责边界：

- `request_parser`
  - 负责解析 JSON / multipart 输入
- `maibao service`
  - 负责调用模型
- `response normalizer`
  - 负责标准化模型返回
- `oss service`
  - 负责上传 OSS

这样以后定位性能问题和做并发优化会容易很多。

## 6. 给每个请求加 requestId

当前协议里已经有 `requestId` 字段，后续建议继续贯彻：

- HTTP 入口记录 `requestId`
- Maibao 调用带 `requestId`
- OSS 上传带 `requestId`
- 错误日志里带 `requestId`

这样以后做链路追踪和问题排查会轻松很多。

## 7. 日志要轻量，避免在高频路径打大对象

后续高频请求里不要打印：

- 原始二进制
- 大段 base64
- 超长响应体
- 整个文件流对象

建议日志只保留：

- `requestId`
- 输入类型
- 模型名
- 文件数量
- 外部调用耗时
- 成功/失败状态

## 8. 为长耗时链路预留异步任务模式

如果后续发现这条链路变长：

- 调模型耗时高
- 文件较大
- 上传 OSS 较慢

那就不要强撑同步接口。

推荐演进方向：

1. HTTP 请求进入后返回 `jobId`
2. 后台 worker 执行：
   - 调模型
   - 上传 OSS
3. 前端或字段捷径轮询结果

短期可以先同步，长期建议预留异步任务能力。

## 9. 预留 Redis / 队列能力

如果后续进入更稳定的生产形态，建议引入：

- `Redis`
  - 做任务状态
  - 做幂等控制
  - 做限流
- 队列/worker
  - `Celery`
  - `RQ`
  - `dramatiq`

当前阶段不用立刻上，但目录结构和接口设计最好不要把自己锁死在纯同步模式。

## 10. 做基础限流和保护

即使是内部调用，也建议加基础保护：

- 单请求文件大小限制
- 单请求文件数量限制
- 请求体大小限制
- 并发任务数限制

因为真正把服务拖垮的往往不是平均请求量，而是异常输入和突发流量。

## 11. 部署形态建议

当前项目适合的演进路线：

### 阶段 1：当前阶段

- Flask
- 单实例
- 生产级进程管理
- 统一超时
- 统一日志

### 阶段 2：小规模生产

- `gunicorn`
- 多 worker
- 反向代理（如 `nginx`）
- requestId 日志串联

### 阶段 3：更稳的生产形态

- 模型调用和上传逻辑异步化
- Redis / 任务队列
- 多副本部署
- 指标监控和告警

## 12. 当前项目后续最值得优先落地的事项

建议按这个顺序推进：

1. backend 真正接入 Maibao 调图
2. backend 真正接入 OSS 上传
3. 所有外部调用加超时
4. 给服务日志统一补 `requestId`
5. 用生产级进程启动 Flask
6. 再考虑异步任务和 Redis

## 一句话总结

对于当前项目，性能优化的关键不是“换语言”或“提前做很重的架构”，而是先把这些基础做好：

- 生产级部署
- 外部请求超时
- 文件流处理
- 清晰服务分层
- requestId 链路追踪
- 为异步任务模式预留空间
